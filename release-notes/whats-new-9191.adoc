== Rede

[cols="30%,70%"]
|===
| Atualização | Descrição 


 a| 
Gerencie as versões NFS suportadas pela SVM
 a| 
O suporte ao NFSv4.2 é gerenciado independentemente das configurações do NFSv4.1, permitindo que cada SVM anuncie ou suprima os recursos do NFSv4.2 conforme necessário. Por padrão, o NFSv4.2 é habilitado quando um novo serviço NFS é criado, a menos que seja explicitamente desabilitado pelo administrador.

* https://docs.netapp.com/us-en/ontap/nfs-admin/enable-disable-nfsv42-task.html["Ativar ou desativar o NFSv4.2 para ONTAP SVMs"]
* https://docs.netapp.com/us-en/ontap/task_nas_enable_linux_nfs.html["Habilitar armazenamento NAS para servidores Linux usando NFS com o ONTAP System Manager"]




 a| 
xref:../concepts/path-failover-concept.html#nas-path-failover[Aprimoramento de failover do LIF]
 a| 
A política de grupo de failover definida pelo sistema ONTAP expande dinamicamente os destinos de failover para LIFs de dados NAS, podendo incluir quaisquer portas disponíveis no domínio de broadcast em qualquer nó disponível no cluster. Você pode aplicar a política definida pelo sistema a qualquer grupo de failover. Antes do ONTAP 9.19.1, as LIFs com essa política podiam realizar failover para apenas um outro nó no cluster; se esse nó estivesse fora do quorum, a LIF não realizaria failover, mesmo que houvesse outros nós disponíveis.

== SAN

[cols="30%,70%"]
|===
| Atualização | Descrição 


 a| 
xref:../san-config/configure-fc-nvme-hosts-ha-pairs-reference.html[Suporte para configurações FC de conexão direta]
 a| 
Não são necessários switches FC para conectar hosts FC e FC-NVMe ao armazenamento ONTAP. Os hosts podem se conectar diretamente às portas de adaptadores FC nos sistemas AFF, ASA e FAS. Essa conexão direta é especialmente útil em locais remotos e ambientes híbridos onde a simplicidade é importante.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Suporte para multipath ativo-ativo em sistemas AFF]
 a| 
Os sistemas AFF suportam multipath ativo-ativo em ambientes SAN. Em configurações ativo-ativo, todos os caminhos dentro do par de HA que detém o LUN ou namespace são otimizados para o modo ativo, de forma que, em caso de failover de armazenamento, o host não precise aguardar a transição ALUA ou ANA dos caminhos de failover para retomar as operações de E/S. Isso reduz o tempo de failover, permitindo que os clientes SAN executem cargas de trabalho críticas em sistemas AFF com a mesma resiliência de caminho sem interrupção e balanceamento de carga anteriormente disponíveis apenas em sistemas ASA.



 a| 
xref:../san-admin/multipath-automatic-lif-failover-support.html[Suporte aprimorado para failover automático de SAN LIF]
 a| 
O failover automático de LIF é suportado nos sistemas AFF e ASA para LIFs de protocolo iSCSI em configurações multipath ativo-ativo e ativo-ativo-somente-scsi, e para LIFs de protocolo NVMe/TCP em configurações multipath ativo-ativo.



 a| 
link:https://docs.netapp.com/us-en/ontap-fli/get-started/learn-about-data-migration.html["Suporte para importação de LUNs externos para o protocolo iSCSI"^]
 a| 
A importação de LUN estrangeira (FLI) suporta o protocolo iSCSI para conectividade de backend.

== Segurança

[cols="30%,70%"]
|===
| Atualização | Descrição 


 a| 
Características da carga de trabalho ARP removidas da interface do usuário do System Manager
 a| 
System Manager não exibe mais as características da carga de trabalho da Proteção Autônoma contra Ransomware (ARP), incluindo as estatísticas de linha de base e de pico. Essas informações foram removidas porque a atividade de pico era frequentemente interpretada erroneamente como um ataque de ransomware real ou como algo que exigia ajuste manual.

O comportamento e a detecção do ARP permanecem inalterados, e apenas a apresentação desses detalhes pelo System Manager é afetada. As estatísticas de surge continuam disponíveis usando as opções da CLI do ONTAP e da API REST para análise avançada.



 a| 
xref:../anti-ransomware/use-cases-restrictions-concept.html#supported-configurations[Suporte para proteção autônoma contra ransomware para SnapMirror síncrono e SnapMirror active sync]
 a| 
O ARP/AI é compatível com volumes que participam de SnapMirror síncronos (NAS e SAN) e de SnapMirror active sync SAN em plataformas suportadas. O ARP/AI pode proteger volumes primários em relacionamentos SnapMirror síncronos em sistemas FAS, AFF e AFX, e volumes primários em relacionamentos SnapMirror active sync SAN em sistemas AFF e ASA r2. O ARP/AI é executado no volume primário ativo de leitura e gravação no relacionamento, criando snapshots ARP somente do primário e fornecendo detecção de ransomware baseada em aprendizado de máquina.



 a| 
xref:../networking/configure-tls-offload.html[Configuração de descarregamento TLS]
 a| 
Você pode configurar o descarregamento de TLS para melhorar o desempenho pós-handshake do TLS, aproveitando os recursos em placas Ethernet compatíveis. Esse recurso descarrega a criptografia e a descriptografia, reduzindo a sobrecarga da CPU e melhorando o desempenho.

== Eficiência de storage

[cols="30%,70%"]
|===
| Atualização | Descrição 


 a| 
xref:../flexcache/index.html[Aprimoramento do desempenho]
 a| 
ONTAP 9.19.1 permite que mais recursos da CPU processem solicitações de listagem de diretórios NAS pequenos. Essa melhoria beneficia clientes com centenas ou milhares de clientes listando simultaneamente o mesmo diretório NAS pequeno, com tamanho de 2 MB ou menos (aproximadamente 25.000 arquivos).

Esse aprimoramento reduz a necessidade de usar soluções alternativas, como volumes locais FlexCache, para suportar altos números de clientes executando listagens de diretórios. Para números menores de clientes, o desempenho não será impactado negativamente. Esse aprimoramento do desempenho é automático e não requer configuração.
